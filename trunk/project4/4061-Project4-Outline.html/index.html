General Requirements
The server will be composed of three different types of threads:
Dispatcher threads
Repeatedly accept an incoming connection, read the request from the connection, and place the request in a queue for a worker thread to pick up and serve, in a loop.
There will only be one request per incoming connection.
Worker threads
Monitor the request queue, pick up requests from it, and serve those requests back to the client.
Put those requests back to another prefetching queue which will be read by the prefetcher threads.
Be careful to synchronize these two bounded queues!
Prefetcher threads
Check if they can guess what the next request is going to be and bring the guessed page from the disk to the cache.
The following functions are provided in pre-compiled object files:
Run once in the main thread;
void init( int port );
These two functions will be used in order, in a loop, in the dispatch threads:
int accept_connection( void );
int get_request( int fd, char *filename );
These functions will be used by the worker threads after handling the request:
int return_result( int fd, char *content_type, char *buf, int numbytes );
int return_error( int fd, char *buf );
For the prefetcher thread:
int nextguess( char *filename, char *guessed );
Full documentation of these functions is in util.h
Incoming Requests
An HTTP request has the form: GET /dir1/subdir1/.../target file HTTP/1.1 where /dir1/subdir1/.../ is assumed to be located under the web tree root location.
The get request function automatically parses this and returns the /dir1/subdir1/.../target file portion in the filename parameter.
The web tree will be rooted at a specific location, specified by one of the arguments to the server. For example, if the web tree is rooted at /home/user/joe/html, then a request for /index.html would map to /home/user/joe/html/index.html.
Caching and Returning Results
The web server will implement caching, a technique that greatly improves performance for frequently fetched pages and files.
This cache stores (request, result) entries, where result is the actual data (in memory) of the file to be returned.
The web cache size (number of entries) is defined by an argument, and new entries may overwrite old ones.
Assume a maximum of 100 cache entries.
If a request is not in the cache, the worker will fetch the file contents from disk and then put the new mapping (request, result) in the cache.
When the file contents are fetched, the file size must also be determined.
When the cache is full, replace the oldest entry in the cache with the new cache entry.
Since multiple threads may be accessing the cache, you must synchronize access to it.
Use return_result to send the data back to the web browser from the worker threads, provided the file was opened and read correctly or the data was found in the cache.
If there was any problem with accessing the file, then you should use return_error instead.
HTTP headers will automatically be added to the data before sending it.
Part of returning the result is sending back a special parameter to the browser: the content-type of the data.
You may make assumptions about the content-type of a file based on its extension:
Files ending in .html or .htm are content-type "text/html"
Files ending in .jpg are content-type "image/jpeg"
Files ending in .gif are content-type "image/gif"
Files that have not been classified in the above categories may be considered, by default, to be of content-type "text/plain"
Prefetching
The prefetcher will try to make a guess for the next file that the client might request for and bring that file to the cache if it is not already there.
The benefit of this if that if the client actually makes that request, the server can very quickly serve it from its cache.
If the prefetching queue is full, worker threads will neither try to add new requests to it nor wait for the queue to be non-full.
Assume that the prefetching queue can store 100 requests.
Worker threads will add new requests only when the prefetching queue is not full.
This is because compared to other tasks the server performs, prefetching is a low priority task.
Prefetcher threads repeatedly check this queue, remove the next request, make a guess and if that guessed file is not already in the cache, bring it to the cache.
Use nextguess to guess the next file based on the current request. This function will return 0 and set the guessed filename if it can find a file that needs to be prefetched. Otherwise it will return a nonzero integer.
If a guess is made, store the guessed file in the cache just like an actual request.
Modes of Operation
First Come First Served (FCFS)
Each request is taken out of the request buffer by the worker in the same order it was inserted by the dispatcher.
Cached Requests First (CRF)
The worker searches the cache to see whether any of the requests can be satisfied from the cache. If there are no cached requests, the server falls back to FCFS.
Smallest File First (SFF)
The smallest file is always taken out of the request queue first by the worker. This is irrespective of whether the file is cached or not.
Use stat system call to get the size of a file.
Request Logging
The worker threads will carefully log each request (normal or error-related) to the file web_server_log in the format below.
[ThreadID#][Request#][fd][Cache HIT/MISS][Request string][bytes/error]
ThreadID# is an integer from 0 to num_workers - 1 indicating the ID of the handling worker thread.
Request# is the total number of requests this specific worker thread has handled so far, including this request
fd is the file descriptor returned by accept_connection() for this request
Cache HIT/MISS is either "HIT" or "MISS" depending on whether this specific request was found in the cache or not
Request string is the filename buffer filled in by the get_request() function
bytes/error is either the number of bytes returned by a successful request or the error string returned by return_error() if an error occurred.
Configuration
port number can be specified
Only ports 1025 - 65535 may be used by default
path is the path to your web root location where all the files will be served from
num_dispatch is how many dispatch threads to start up
num_workers is how many worker threads to start up
num_prefetch is how many prefetch threads to start up
qlen is the fixed, bounded length of the request queue
mode can be FCFS (value=0) or CRF (value=1) or SFF (value=2)
FCFS: First Come First Served
CRF: Cached Requests First
SFF: Smallest File First
cache-entries is the number of entries available in the cache
The server should be run as:
% web_server_http port path num_dispatch num_workers num_prefetch qlen mode cache_entries
